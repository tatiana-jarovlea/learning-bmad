# Story 6.5: Smoke Testing & Production Deploy

**Epic:** 6 — i18n, Polish & Launch Readiness
**Status:** Draft

---

## Story

**As** the PawTrust team,
**I want** a verified production environment with documented smoke tests,
**so that** we can confidently launch the platform knowing all critical user paths work end-to-end.

---

## Acceptance Criteria

1. A `docs/smoke-tests.md` document exists with exactly 5 critical user paths documented as step-by-step manual test scripts.
2. All 5 smoke tests pass on the staging environment before the production deploy is initiated.
3. Production infrastructure is provisioned: EC2/Elastic Beanstalk (backend), RDS MySQL 8 (`eu-central-1`), S3 prod buckets (documents, photos), CloudFront distribution (frontend), SES production domain verified (`@pawtrust.md`), Stripe live keys configured.
4. All database migrations run successfully on production RDS; the initial admin user is seeded.
5. DNS is configured: primary domain (`pawtrust.md` + `www.pawtrust.md`) → CloudFront (frontend SPA); API subdomain (`api.pawtrust.md`) → Elastic Beanstalk backend.
6. Post-deploy checklist verified on production: HTTPS active on all domains, emails flowing from SES, Stripe live mode responds correctly, all 5 smoke tests pass on production.
7. Rollback plan is documented in `docs/runbook/rollback.md`.

## Tasks / Subtasks

### Documentation

- [ ] **T1** — Create `docs/smoke-tests.md`

  Document the following 5 critical user paths as numbered, step-by-step manual test scripts. Each test must specify: preconditions, steps, expected result, and pass criteria.

  **Path 1 — Buyer full flow: Register → Browse → Inquire → Contact revealed**
  - Preconditions: Clean test email address; at least one active listing exists on the platform
  - Steps:
    1. Navigate to `https://pawtrust.md`, click Register
    2. Register as Buyer with test email; verify email arrives in inbox
    3. Click verification link; confirm redirect to dashboard
    4. Browse listings page; open a listing detail
    5. Click "Make Inquiry" — submit inquiry form
    6. Confirm response: inquiry submitted, contact not yet revealed
    7. Navigate back to listing — confirm contact (phone/email) is now visible
  - Pass criteria: Contact details rendered, no console errors, no 5xx responses

  **Path 2 — Breeder full flow: Register → Create listing → Pay via Stripe → Listing visible in search**
  - Preconditions: Clean test email; Stripe test card `4242 4242 4242 4242`
  - Steps:
    1. Register as Breeder; verify email; complete profile
    2. Create a new listing (fill all required fields, upload a photo)
    3. Proceed to Stripe Checkout; use test card
    4. Confirm redirect to success page; listing status = active
    5. Navigate to listings search as an unauthenticated user; confirm listing appears
  - Pass criteria: Listing visible in public search, Stripe payment recorded, no errors

  **Path 3 — Verification flow: Breeder submits → Admin approves → Verified badge appears**
  - Preconditions: Breeder account with profile created; admin account available
  - Steps:
    1. As Breeder: navigate to dashboard → Verification tab
    2. Upload required documents; submit verification request
    3. Confirm notification email received by breeder
    4. Log in as Admin; navigate to verification queue
    5. Open the verification request; approve it
    6. Log out as Admin; browse to the breeder's public profile
    7. Confirm "Verified" badge is visible
  - Pass criteria: Badge visible publicly, breeder receives approval email

  **Path 4 — Sitter flow: Register → Create profile → Appears on discovery page**
  - Preconditions: Clean test email; no Stripe required (if `SKIP_LISTING_PAYMENT=false` on prod, use test card)
  - Steps:
    1. Register as Pet Sitter; verify email
    2. Navigate to Sitter Dashboard → Create Profile
    3. Fill out profile details (name, bio, services, service area, rate, photo)
    4. Save profile
    5. Navigate to `/sitters` (Sitter Discovery page) as unauthenticated user
    6. Confirm new sitter profile appears in results
  - Pass criteria: Sitter visible on discovery page within 30 seconds of profile save

  **Path 5 — Admin access control: Non-admin cannot access admin dashboard**
  - Preconditions: Buyer account; Admin account
  - Steps:
    1. Log in as Buyer
    2. Attempt to navigate to `/admin/dashboard` directly
    3. Confirm: 403 Forbidden or redirect to home page
    4. Log in as Admin
    5. Navigate to `/admin/dashboard`
    6. Confirm: dashboard loads, shows pending verifications count
  - Pass criteria: Buyer receives 403/redirect; Admin sees dashboard without errors

- [ ] **T2** — Create `docs/runbook/rollback.md`

  Document the rollback plan with the following sections:

  **§1 — When to Roll Back**
  - Any smoke test fails on production after deploy
  - Error rate on Sentry/CloudWatch > 5% of requests in 30 minutes post-deploy
  - Payment processing broken (Stripe not responding, webhook failing)
  - Database migration failure

  **§2 — Frontend Rollback**
  - CloudFront serves the previous S3 deployment (keep the previous build in S3)
  - Procedure: In AWS Console → CloudFront → Invalidate `/*` → Point distribution to previous S3 folder
  - Or via CLI:
    ```bash
    aws s3 sync s3://pawtrust-frontend-prod-previous/ s3://pawtrust-frontend-prod/
    aws cloudfront create-invalidation --distribution-id $DIST_ID --paths "/*"
    ```

  **§3 — Backend Rollback**
  - Elastic Beanstalk environment has built-in rollback to previous version
  - Procedure: AWS Console → Elastic Beanstalk → Environment → Application Versions → Deploy previous version
  - Or via CLI:
    ```bash
    eb deploy --version <previous-version-label>
    ```

  **§4 — Database Rollback**
  - If a migration caused data issues: restore from RDS automated snapshot (taken before migrations ran)
  - Pre-migration snapshot is mandatory — tag it: `pre-deploy-$(date +%Y%m%d)`
  - To restore: create a new RDS instance from the snapshot; update `.env` `DB_HOST` to point to restored instance

  **§5 — Rollback Communication**
  - Notify team in Slack `#pawtrust-ops` immediately
  - Document the incident in `docs/runbook/incidents/YYYY-MM-DD.md`
  - Do not re-deploy until root cause is identified and fixed

- [ ] **T5** — Seed initial admin user on production

  ```bash
  # On the EB instance
  php artisan db:seed --class=ProductionAdminSeeder --force
  ```

  `ProductionAdminSeeder` must:
  - Read `ADMIN_EMAIL` and `ADMIN_PASSWORD` from environment variables (NOT hardcoded)
  - Create or update the admin user with role `admin`
  - Log the seeding action to `AdminAuditLog`
  - Be idempotent (uses `firstOrCreate` — safe to run twice)

  ```php
  // database/seeders/ProductionAdminSeeder.php
  class ProductionAdminSeeder extends Seeder
  {
      public function run(): void
      {
          $email = env('ADMIN_EMAIL');
          $password = env('ADMIN_PASSWORD');

          if (!$email || !$password) {
              $this->command->error('ADMIN_EMAIL and ADMIN_PASSWORD env vars required.');
              return;
          }

          $user = User::firstOrCreate(
              ['email' => $email],
              [
                  'name' => 'PawTrust Admin',
                  'password' => Hash::make($password),
                  'email_verified_at' => now(),
                  'locale' => 'ro',
              ]
          );

          $user->syncRoles(['admin']);
      }
  }
  ```

  - After seeding: immediately rotate the `ADMIN_PASSWORD` env var to remove it from the environment (set to empty string or remove from EB config); admin should change password on first login
  - **⚠️ `env()` and config cache:** `env()` returns `null` after `php artisan config:cache` is run. Run this seeder BEFORE `config:cache`, or clear the cache first with `php artisan config:clear`, then re-cache after seeding:
    ```bash
    php artisan config:clear
    php artisan db:seed --class=ProductionAdminSeeder --force
    php artisan config:cache
    ```

## Dev Notes

### §1 — Environment Variables Security

Production environment variables are set in the Elastic Beanstalk console, NOT in a `.env` file checked into git. The `.env` file in the repo is `.env.example` only. The `.gitignore` must include `.env` — verify this before deploy.

The `ADMIN_PASSWORD` env var is set temporarily for the seeding step only and removed immediately after. This prevents it from persisting in the environment indefinitely.

---

### §2 — CloudFront SPA Routing

The CloudFront distribution must be configured with a custom error page for 404 responses from S3:

- Error code: 404
- Response page: `/index.html`
- Response code: 200

Without this, direct navigation to deep links (e.g., `https://pawtrust.md/listings/123`) will return a 403/404 from S3 instead of serving the React SPA.

---

### §3 — Staging vs Production Stripe Keys

Staging uses Stripe test keys (`pk_test_`, `sk_test_`). Production uses live keys (`pk_live_`, `sk_live_`). The Stripe Checkout page will display "TEST MODE" in a banner if test keys are active. This banner's absence on production is part of the post-deploy checklist.

The live webhook endpoint must be registered separately from the test webhook — Stripe's dashboard has separate sections for Test and Live webhook endpoints.

---

### §4 — Database Migration Safety

Laravel migrations run in order; running `migrate --force` on a populated production database is irreversible for destructive migrations (e.g., `dropColumn`, `dropTable`). For this initial deploy all migrations are additive (fresh schema). Future deploys must be reviewed for destructive operations.

Always take a manual RDS snapshot before running migrations. The automated backup alone is insufficient because it may run at a scheduled time — not immediately pre-deploy.

---

### §5 — What Happens After This Story

This is the final story of the PawTrust MVP. After a successful production deploy:

1. Monitor Sentry (or CloudWatch Logs) for errors in the first 24 hours
2. Send the smoke test results to stakeholders as sign-off confirmation
3. Schedule a post-launch retrospective
4. Begin the post-MVP backlog:
   - External penetration test
   - WAF implementation
   - SSR / Next.js migration for SEO
   - 2FA for admin accounts
   - Scaling plan (RDS read replica, EB auto-scaling policy)
