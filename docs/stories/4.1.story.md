# Story 4.1: Inquiry Submission

**Epic:** 4 — Inquiry System & Contact Reveal
**Status:** Approved

---

## Story

**As a** buyer,
**I want** to submit an inquiry on a listing,
**so that** I can contact the breeder and access their documents.

---

## Acceptance Criteria

1. Inquiry submission endpoint: `POST /api/v1/listings/{id}/inquire`. Auth required, buyer role only. Breeders cannot inquire on their own or any other listing — return 403.
2. Request body: optional `message` (max 500 chars).
3. Creates an inquiry record with: `listing_id`, `buyer_id`, `message`, `status`, `created_at`.
4. One inquiry per buyer per listing enforced. Duplicate attempt returns 409 with `{ message: "You have already inquired about this listing." }`.
5. On successful inquiry: response includes `{ inquiry_id, breeder_contact: { phone, email, name } }`. This is the **only** API endpoint that returns breeder contact details. Contact fields are never exposed on any other endpoint.
6. Inquiry `status` is automatically set to `contact_revealed` on successful submission — no additional step required from buyer or breeder.
7. Can be verified locally: create listing, submit inquiry, verify contact details returned, verify second attempt returns 409.

    ```php
    public function submit(User $buyer, Listing $listing, ?string $message): Inquiry
    {
        // 1. Breeders cannot inquire — defensive layer behind route middleware
        if ($buyer->hasRole('breeder')) {
            abort(403, 'Breeders cannot submit inquiries.');
        }

        // 2. Check listing is active
        if ($listing->status !== 'active') {
            abort(422, 'This listing is no longer active.');
        }

        // 3. Check duplicate
        $existing = Inquiry::where('listing_id', $listing->id)
            ->where('buyer_id', $buyer->id)
            ->first();

        if ($existing) {
            throw new DuplicateInquiryException('You have already inquired about this listing.');
        }

        // 4. Create inquiry — status auto-set to contact_revealed
        return Inquiry::create([
            'listing_id' => $listing->id,
            'buyer_id'   => $buyer->id,
            'message'    => $message,
            'status'     => 'contact_revealed',
        ]);
    }
    ```

  - `DuplicateInquiryException` — custom exception extending `RuntimeException` with HTTP status 409
  - The exception handler in `app/Exceptions/Handler.php` must map it to a 409 JSON response

### §1 — Contact Data Security Boundary

**Breeder contact details (`phone`, `email`) are NEVER returned by any API endpoint except `POST /api/v1/listings/{id}/inquire`.**

This must be enforced at the resource layer:

- `BreederProfileResource` — no `phone`, no `email` (enforced in Story 2.1, verified here by regression test)
- `ListingResource` — no `breeder.phone`, no `breeder.email`
- `AdminVerificationRequestResource` — `email` is present for admin use only (Story 3.2, documented restriction)

The two regression tests `test_phone_and_email_not_present_in_listing_api_response()` and `test_phone_and_email_not_present_on_public_breeder_profile()` serve as permanent guards — they must remain in the test suite indefinitely.

---

### §2 — Why `status = contact_revealed` Immediately

The MVP design skips any intermediate "pending" state where the breeder would need to accept/reject the inquiry before contact is revealed. This is a deliberate simplification:

- Moldova/Romania market: buyers expect immediate contact access, similar to classifieds platforms (OLX, Storia)
- The breeder is notified (Story 4.2) and can ignore/follow up at their discretion
- A future "request acceptance" flow can be added post-MVP by adding a `breeder_accepted_at` field and a breeder-side accept endpoint, without breaking existing inquiry records

The `status` column is retained for extensibility — it starts at `contact_revealed` instead of `pending` for MVP to keep the flow simple.

---

### §3 — Breeder Role Guard — Two Layers

1. **Route middleware** `role:buyer` (Spatie) — returns 403 before the controller is reached
2. **Service check** `if ($buyer->hasRole('breeder'))` — defensive second layer in case middleware is misconfigured

Both layers must be tested independently (route-level 403 test + service unit test if applicable).
