# Story 5.2: Pet Sitter Discovery Page

**Epic:** 5 — Pet Sitter Profiles & Pay-per-Listing
**Status:** Draft

---

## Story

**As a** buyer or pet owner,
**I want** to browse pet sitters by location and service type,
**so that** I can find local care for my pet.

---

## Acceptance Criteria

1. Public endpoint `GET /api/v1/sitters` — paginated list (20/page) of sitter cards: full name, service areas, services offered, rates, first portfolio photo (or placeholder), bio excerpt (first 150 chars), average review score (return `null` for now — review system is post-MVP).
2. Filterable by: `service_area` (partial string match on any service area), `service` (exact match on services array: `boarding`, `dog_walking`, `day_care`, `drop_in`).
3. Default sort: most recently updated profile first (`updated_at DESC`).
4. No authentication required for browsing.
5. Frontend route `/sitters` — filter sidebar (service area text input + service type checkboxes), responsive card grid, each card links to `/sitters/{id}`.
6. Empty state message when no results: "No sitters found in [location]. Try expanding your area."

- [ ] **T2** — Create `SitterDiscoveryService`
  - File: `app/Services/SitterDiscoveryService.php`
  - Method: `list(array $filters): LengthAwarePaginator`

  ```php
  public function list(array $filters): LengthAwarePaginator
  {
      $query = SitterProfile::with(['portfolioPhotos' => fn($q) => $q->orderBy('sort_order')->limit(1)])
          ->latest('updated_at');

      if (!empty($filters['service_area'])) {
          // JSON contains search — works in MySQL 8 with JSON_CONTAINS or LIKE on JSON column
          $area = '%' . $filters['service_area'] . '%';
          $query->where('service_areas', 'like', $area);
      }

      if (!empty($filters['service'])) {
          $query->whereJsonContains('services', $filters['service']);
      }

      return $query->paginate(20);
  }
  ```

  - Note: `whereJsonContains` for `services` uses MySQL JSON functions — requires MySQL 8 (already the project DB). For `service_area`, a LIKE on the JSON column is a pragmatic MVP approach; for production scale, consider a `sitter_service_areas` pivot table. Document as known technical debt.

## Dev Notes

### §1 — JSON Column Filtering Performance

The `service_areas` filter uses `LIKE '%query%'` on a JSON column. This is acceptable at MVP with a small number of sitter profiles but does NOT use an index — it will full-scan the `sitter_profiles` table.

If the sitter count grows significantly, this should be migrated to a `sitter_service_areas` pivot table:

```
sitter_service_areas: id | sitter_profile_id | city | region
```

with a standard index on `city`. The `whereJsonContains` for `services` has better MySQL native support via functional indexes but is still a full-scan without additional setup.

**This is explicitly documented as known technical debt** — it does not block MVP. Add a `// TODO: normalize service_areas to pivot table at scale` comment in `SitterDiscoveryService`.

---

### §2 — Average Review Placeholder

`average_review: null` is returned by the API. The frontend renders "-" for null. This field is included in the contract now so that when the review system is added (post-MVP), no frontend changes are needed for the sitter card — only the backend needs to start returning the actual value.

Do not add a `reviews` relationship to `SitterProfile` in this story — it does not exist yet. The `null` value is hardcoded in `SitterCardResource`.

---

### §3 — Photo in Sitter Card

The sitter card shows the first portfolio photo (sort_order = 0). If no portfolio photos exist, `photo_url` is `null` — the frontend renders a default placeholder SVG. Do NOT make an S3 call to generate a pre-signed URL for the card. Portfolio photos are served via CloudFront (public URL derived from `s3_key`) — not pre-signed. This is the same as listing photos on the browse page.

Confirm that `SitterPortfolioPhoto::getPhotoUrlAttribute()` returns a CloudFront URL (Story 5.1 T5), not a pre-signed S3 URL, since these are portfolio/marketing images that are meant to be publicly accessible (not private medical documents).
