# Story 1.5: Admin User Seeding

**Epic:** 1 ŌĆö Foundation & Authentication
**Status:** Draft

## Story

**As a** developer,
**I want** a seeder that creates an initial admin user from environment variables,
**so that** the platform has an admin account for testing and launch without hardcoding credentials.

---

## Acceptance Criteria

1. Laravel seeder `AdminUserSeeder` creates one admin user with credentials read from `.env`: `ADMIN_EMAIL`, `ADMIN_PASSWORD`.
2. Admin role is one of the valid `role` enum values on the `users` table ŌĆö the migration from Story 1.2 must be updated to include `admin` in the enum if it does not already. Consistent with Spatie roles from Stories 1.2ŌĆō1.3.
3. `php artisan db:seed --class=AdminUserSeeder` runs without error on a fresh database.
4. Admin user can log in via `POST /api/v1/auth/login` and receives a Sanctum token where `user.role === 'admin'`.
5. Seeder is idempotent ŌĆö running it twice does not create a duplicate admin user (`firstOrCreate` on email).
6. `ADMIN_EMAIL` and `ADMIN_PASSWORD` added to `.env.example` with placeholder values and a comment warning against committing real credentials.
7. Seeding step documented in `README.md` of `pawtrust-backend`.

  ```php
  <?php

  namespace Database\Seeders;

  use App\Models\User;
  use Illuminate\Database\Seeder;
  use Illuminate\Support\Facades\Hash;

  class AdminUserSeeder extends Seeder
  {
      public function run(): void
      {
          $email    = config('app.admin_email');
          $password = config('app.admin_password');

          if (empty($email) || empty($password)) {
              $this->command->warn('ADMIN_EMAIL or ADMIN_PASSWORD not set ŌĆö skipping AdminUserSeeder.');
              return;
          }

          $user = User::firstOrCreate(
              ['email' => $email],
              [
                  'name'     => 'Admin',
                  'password' => Hash::make($password),
                  'role'     => 'admin',
                  'locale'   => 'ro',
              ]
          );

          // Assign Spatie role (idempotent ŌĆö syncRoles replaces existing)
          $user->syncRoles(['admin']);

          $this->command->info("Admin user ready: {$email}");
      }
  }
  ```

  - Reads credentials from `config()` not directly from `env()` ŌĆö follows Laravel config caching best practice
  - Warns and exits gracefully if env vars are not set ŌĆö prevents silent failure in CI

## Dev Notes

### ┬¦1 ŌĆö Why `config()` Instead of `env()`

Laravel caches config in production with `php artisan config:cache`. After caching, direct `env()` calls in application code return `null`. Reading through `config('app.admin_email')` is the correct pattern ŌĆö it works whether or not config is cached.

The seeder runs via `php artisan`, which does NOT use the config cache by default. However, using `config()` is still the correct habit and ensures the seeder works if called from a context where config is cached (e.g., post-deployment scripts).

---

### ┬¦1b ŌĆö Spatie Guard Name

Spatie Laravel Permission associates roles with a guard name. The default is `'web'`. With Sanctum API-only apps, there are two valid approaches ŌĆö **pick one and apply consistently**:

**Option A (recommended for this project):** Keep Spatie using `'web'` guard (the default). Add to `app/Models/User.php`:

```php
protected string $guard_name = 'web';
```

And create roles with `'guard_name' => 'web'` (or omit it ŌĆö Spatie defaults to `'web'`).

**Option B:** Configure Spatie to use `'sanctum'` guard globally in `config/permission.php`:

```php
'guard' => 'sanctum',
```

Then create roles with `'guard_name' => 'sanctum'`.

**The T2 code uses `'sanctum'` ŌĆö this requires Option B to be in place.** If `config/permission.php` still has the default `'web'` guard, `$user->hasRole('admin')` will silently return `false`. Resolve this before Story 2.1 uses any role checks.

> Recommendation: Use **Option A** (`'web'` guard) ŌĆö it requires less configuration and is more robust with Laravel's default auth stack.

---

### ┬¦2 ŌĆö Idempotency

`User::firstOrCreate(['email' => $email], [...])` only creates the user if no user with that email exists. Running the seeder twice:

- First run: creates the user, assigns role
- Second run: finds existing user, re-calls `syncRoles(['admin'])` (harmless)

`syncRoles` replaces all current Spatie roles ŌĆö it does NOT accumulate. This is the correct behaviour for a seeder.

---

### ┬¦3 ŌĆö Enum vs Spatie Roles

The `users.role` column (plain enum) and Spatie's `roles` table serve different purposes:

|         | `users.role` (enum)                 | Spatie roles table             |
| ------- | ----------------------------------- | ------------------------------ |
| Purpose | Quick lookup, stored in user row    | Fine-grained permission checks |
| Used by | `UserResource`, frontend auth store | `$user->hasRole()`, policies   |
| Set at  | Registration / seeder               | `syncRoles()` calls            |

Both must be set correctly. The `users.role` enum value drives what dashboard the user sees (frontend). The Spatie role drives what API actions they can perform (middleware/policies in later epics). They must always be kept in sync ŌĆö the service layer is responsible for this.
