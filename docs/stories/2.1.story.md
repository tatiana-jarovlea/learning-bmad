# Story 2.1: Breeder Profile Creation & Public View

**Epic:** 2 — Breeder Profiles & Listings
**Status:** Done

---

## Story

**As a** breeder,
**I want** to create and manage my public profile,
**so that** buyers can learn about my kennel and credentials before contacting me.

---

## Acceptance Criteria

1. Breeder can fill in profile fields: kennel name, description (max 1000 chars), location (city + region — dropdown from defined Moldova/Romania city list), years active, breed specialization (multi-select from seed breed list), profile photo (upload to S3).
2. Profile saved via `POST /api/v1/breeder/profile` (create) and `PUT /api/v1/breeder/profile` (update). Requires `auth:sanctum` + breeder role.
3. Public profile accessible at `GET /api/v1/breeders/{id}` (no auth required). Returns: kennel name, description, location, years active, breed specializations, profile photo URL (CloudFront pre-signed, 24h TTL), all active listings (summary), verification status. Does NOT return phone, email, or any contact details.
4. Breeder contact details (`phone`, `email`) are never included in the public profile API response.
5. Breeder dashboard shows a "Complete your profile" banner when required fields (kennel_name, description, location) are missing.
6. Can be verified locally: create profile via API → fetch public profile → confirm no contact fields in response.

## Dev Notes

### §1 — Contact Data Security: Explicit Exclusion Pattern

`BreederProfilePublicResource` must NEVER include `phone`, `email`, or any contact fields. This is critical to the platform's trust model — contact reveal is gated behind an accepted, paid inquiry.

The resource file must include a comment:

```php
// SECURITY: Contact details intentionally excluded.
// Access to phone/email is gated via POST /api/v1/inquiries → acceptance flow.
// See: architecture/security.md §Contact Gating
```

A future engineer should never add `phone` to this resource without understanding the business model. The comment prevents accidental exposure.

### §2 — Backend Feature Test Structure

```php
<?php

namespace Tests\Feature\BreederProfile;

use Tests\TestCase;
use Illuminate\Foundation\Testing\RefreshDatabase;
use Illuminate\Http\UploadedFile;
use Illuminate\Support\Facades\Storage;
use App\Models\User;
use App\Models\BreederProfile;

class BreederProfileTest extends TestCase
{
    use RefreshDatabase;

    public function test_breeder_can_create_profile(): void
    {
        $breeder = User::factory()->create(['role' => 'breeder']);
        $breeder->assignRole('breeder');

        $this->actingAs($breeder, 'sanctum')
            ->postJson('/api/v1/breeder/profile', [
                'kennel_name' => 'Golden Paws Kennel',
                'location'    => 'Chișinău, Moldova',
                'description' => 'Specializing in Golden Retrievers.',
            ])
            ->assertStatus(201)
            ->assertJsonPath('data.kennel_name', 'Golden Paws Kennel');

        $this->assertDatabaseHas('breeder_profiles', ['user_id' => $breeder->id]);
    }

    public function test_buyer_cannot_create_breeder_profile(): void
    {
        $buyer = User::factory()->create(['role' => 'buyer']);
        $buyer->assignRole('buyer');

        $this->actingAs($buyer, 'sanctum')
            ->postJson('/api/v1/breeder/profile', ['kennel_name' => 'Test'])
            ->assertStatus(403);
    }

    public function test_public_profile_excludes_phone_and_email(): void
    {
        $breeder = User::factory()->create(['role' => 'breeder']);
        BreederProfile::factory()->create([
            'user_id' => $breeder->id,
            'phone'   => '+37360123456',
        ]);

        $this->getJson("/api/v1/breeders/{$breeder->id}")
            ->assertStatus(200)
            ->assertJsonMissing(['phone'])
            ->assertJsonMissing(['email']);
    }

    public function test_profile_photo_upload_stores_s3_key(): void
    {
        Storage::fake('s3');
        $breeder = User::factory()->create(['role' => 'breeder']);
        $breeder->assignRole('breeder');
        $profile = BreederProfile::factory()->create(['user_id' => $breeder->id]);

        $file = UploadedFile::fake()->image('photo.jpg', 800, 600);

        $this->actingAs($breeder, 'sanctum')
            ->postJson('/api/v1/breeder/profile/photo', ['profile_photo' => $file])
            ->assertStatus(200);

        $this->assertNotNull($profile->fresh()->profile_photo_key);
        Storage::disk('s3')->assertExists($profile->fresh()->profile_photo_key);
    }
```

### §3 — Breeds: Table vs Config File

The breed list is stored in a `breeds` DB table (via `BreedsSeeder`) rather than a PHP config file because:

1. Bilingual (name_ro / name_ru) — easier to query by locale
2. Future admin UI can add breeds without a deploy
3. Filtered by `species` via a simple query

The API endpoint `GET /api/v1/breeds?species=dog` is uncached at MVP — response is small (~30 breeds). Add response caching in a future optimization story if needed.

---

### §4 — Location: String vs Structured

Location is stored as a single `varchar(255)` string (e.g. `"Chișinău, Moldova"`) rather than separate city/region columns. The PRD requires a dropdown UI with a defined city list — the selected value is formatted as a string before saving. This keeps the data model simple for MVP while still enabling frontend-driven filtering in Story 2.4 (which uses `location_city` as a query parameter, parsed from the string server-side).
