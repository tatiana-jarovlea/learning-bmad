# Story 3.4: Verification Status Email Notifications

**Epic:** 3 — Verification System
**Status:** Done

---

## Story

**As a** breeder or admin,
**I want** to receive email notifications when a verification request changes status,
**so that** I am always informed of the progress without having to poll the dashboard.

---

## Acceptance Criteria

1. When a breeder submits a verification request, an email is sent to the admin team at the address configured in `ADMIN_NOTIFICATION_EMAIL`.
2. When an admin sets status to `under_review`, the breeder receives an email confirming their request is being reviewed.
3. When an admin approves the request, the breeder receives a congratulations email with a direct link to their public profile.
4. When an admin rejects the request, the breeder receives an email including the rejection notes and a call-to-action to resubmit.
5. All emails are dispatched via **Laravel Queue** (async) — no synchronous mail sending in any controller or service.
6. Email content is **bilingual (RO/RU)** — rendered in the breeder's locale (`users.locale`) or the application default locale.
7. Local development and test environments use the `log` mail driver to avoid sending real emails.

- [ ] **T4** — Create `VerificationNotificationService`
  - File: `app/Services/VerificationNotificationService.php`
  - Responsibility: dispatch the correct Mailable to the correct recipient(s) for each event
  - Methods:

    ```php
    public function notifyAdminNewRequest(VerificationRequest $vr): void
    {
        Mail::to(config('app.admin_notification_email'))
            ->queue(new AdminNewVerificationRequestMail($vr->breederProfile));
    }

    public function notifyBreederUnderReview(VerificationRequest $vr): void
    {
        $user = $vr->breederProfile->user;
        Mail::to($user->email)
            ->locale($user->locale)
            ->queue(new BreederVerificationUnderReviewMail($vr->breederProfile));
    }

    public function notifyBreederApproved(VerificationRequest $vr): void
    {
        $user = $vr->breederProfile->user;
        Mail::to($user->email)
            ->locale($user->locale)
            ->queue(new BreederVerificationApprovedMail($vr->breederProfile));
    }

    public function notifyBreederRejected(VerificationRequest $vr): void
    {
        $user = $vr->breederProfile->user;
        Mail::to($user->email)
            ->locale($user->locale)
            ->queue(new BreederVerificationRejectedMail($vr->breederProfile, $vr->admin_notes));
    }
    ```

### §1 — Why Notifications Are Outside the DB Transaction

Both the `VerificationRequestService::submit()` and `AdminVerificationService::review()` methods have DB transactions or sequential DB operations. Email dispatch (`Mail::queue()`) should happen AFTER the database write is committed — if the queue dispatch fails, the core business operation (creating/updating the verification request) should not roll back. Conversely, if the DB write fails and an exception is thrown, the email is never dispatched (since we're past the `try/catch` or the transaction prevented the code from reaching the dispatch call).

This pattern mirrors the audit log decision in Story 3.2 §1.

---

### §2 — Locale Handling in Queue Workers

Queue workers run in their own process. The `app.locale` of the queued job is set at dispatch time by `Mail::to(...)->locale($user->locale)->queue(...)`. Laravel serializes the locale into the queued job payload. This means the Blade template renders in the correct locale even when the worker's default locale differs.

Do NOT rely on `App::setLocale()` inside Mailable constructors — the constructor runs at dispatch time in the web process, but locale-sensitive rendering happens later in the worker.

Use `Lang::get('mail.verification.approved.subject', $params, $this->locale)` in the `envelope()` subject if you need to call translations inside the Mailable class itself.

### §4 — SES Configuration (Production)

For production, the following `.env` / secrets manager values are needed (do NOT add to `.env.example` with real values):

```
MAIL_MAILER=ses
AWS_SES_KEY=...
AWS_SES_SECRET=...
AWS_SES_REGION=eu-central-1
MAIL_FROM_ADDRESS=noreply@pawtrust.md
MAIL_FROM_NAME="PawTrust"
QUEUE_CONNECTION=sqs
SQS_PREFIX=...
SQS_QUEUE=pawtrust-notifications
```

This story only implements the Mailable + queue plumbing. SES credential provisioning is an infrastructure task (Story 1.1 / DevOps — already noted in architecture).
