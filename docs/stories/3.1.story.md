# Story 3.1: Breeder Verification Request Submission

**Epic:** 3 — Verification System
**Status:** Approved

---

## Story

**As a** breeder,
**I want** to submit a verification request with my uploaded documents,
**so that** I can earn a Verified badge and build buyer trust.

---

## Acceptance Criteria

1. Verification request submitted via `POST /api/v1/verification-requests`. Auth required, breeder role only.
2. Request payload: array of `document_ids` to include (must belong to the authenticated breeder's profile, and have `status = 'pending'` or `'approved'`; soft-deleted documents rejected).
3. Submission creates a `verification_requests` record with `status = 'pending'`. Breeders cannot submit a new request while one is `pending` or `under_review` — returns 422 with appropriate message.
4. Breeder dashboard shows current verification status: `Not Submitted`, `Pending Review`, `Under Review`, `Verified`, `Rejected (with admin notes)`.
5. On rejection, admin notes visible to breeder in the dashboard. Breeder can resubmit by creating a new verification record — history is preserved; old records are not overwritten.
6. Rejected breeders retain `Unverified` status and remain able to list pets normally.
7. Each submission stored as a separate record — full audit trail of all attempts.

    ```php
    public function submit(User $user, array $documentIds): VerificationRequest
    {
        $profile = $user->breederProfile;

        // Block if active submission exists
        $active = $profile->verificationRequests()
            ->whereIn('status', ['pending', 'under_review'])
            ->first();

        if ($active) {
            throw new \DomainException(
                'A verification request is already ' . $active->status . '. Please wait for a decision.'
            );
        }

        $request = VerificationRequest::create([
            'breeder_profile_id' => $profile->id,
            'status'             => 'pending',
        ]);

        $request->documents()->sync($documentIds);

        return $request;
    }
    ```

    ```php
    public function getStatusForBreeder(BreederProfile $profile): string
    {
        if ($profile->verified_at) return 'verified';

        $latest = $profile->verificationRequests()->latest()->first();
        if (!$latest)                              return 'not_submitted';
        if ($latest->status === 'pending')         return 'pending';
        if ($latest->status === 'under_review')    return 'under_review';
        if ($latest->status === 'rejected')        return 'rejected';
        return 'not_submitted';
    }
    ```

## Dev Notes

### §1 — Status Transition Rules

| Current Status          | Can Submit New Request?          |
| ----------------------- | -------------------------------- |
| `not_submitted`         | ✅ Yes                           |
| `pending`               | ❌ No — "Already pending"        |
| `under_review`          | ❌ No — "Under review"           |
| `approved` (= verified) | ❌ Not needed — already verified |
| `rejected`              | ✅ Yes — can resubmit            |

The `isPending()` helper on the model checks for both `pending` and `under_review` in a single call. The service checks for these before creating a new record.

On resubmission after rejection: a new `verification_requests` record is created; the old rejected record is NOT updated. This preserves full audit history of every submission, its included documents, admin decision, and notes.

### §2 — Backend Feature Test Structure

```php
<?php

namespace Tests\Feature\Verification;

use Tests\TestCase;
use Illuminate\Foundation\Testing\RefreshDatabase;
use App\Models\User;
use App\Models\BreederProfile;
use App\Models\BreederDocument;
use App\Models\VerificationRequest;

class VerificationSubmissionTest extends TestCase
{
    use RefreshDatabase;

    private User $breeder;
    private BreederProfile $profile;
    private BreederDocument $doc;

    protected function setUp(): void
    {
        parent::setUp();
        $this->breeder = User::factory()->create(['role' => 'breeder']);
        $this->breeder->assignRole('breeder');
        $this->profile = BreederProfile::factory()->create(['user_id' => $this->breeder->id]);
        $this->doc = BreederDocument::factory()->create([
            'breeder_profile_id' => $this->profile->id,
            'status'             => 'pending',
        ]);
    }

    public function test_breeder_can_submit_verification_request(): void
    {
        $this->actingAs($this->breeder, 'sanctum')
            ->postJson('/api/v1/verification-requests', [
                'document_ids' => [$this->doc->id],
            ])
            ->assertStatus(201)
            ->assertJsonPath('data.status', 'pending');

        $this->assertDatabaseHas('verification_requests', [
            'breeder_profile_id' => $this->profile->id,
            'status'             => 'pending',
        ]);
    }

    public function test_cannot_resubmit_while_pending(): void
    {
        VerificationRequest::factory()->create([
            'breeder_profile_id' => $this->profile->id,
            'status'             => 'pending',
        ]);

        $this->actingAs($this->breeder, 'sanctum')
            ->postJson('/api/v1/verification-requests', [
                'document_ids' => [$this->doc->id],
            ])
            ->assertStatus(422);
    }

    public function test_can_resubmit_after_rejection(): void
    {
        VerificationRequest::factory()->create([
            'breeder_profile_id' => $this->profile->id,
            'status'             => 'rejected',
            'admin_notes'        => 'Documents illegible. Please resubmit.',
        ]);

        $this->actingAs($this->breeder, 'sanctum')
            ->postJson('/api/v1/verification-requests', [
                'document_ids' => [$this->doc->id],
            ])
            ->assertStatus(201);

        // Both records preserved
        $this->assertDatabaseCount('verification_requests', 2);
    }

    public function test_cannot_submit_with_another_breeders_document(): void
    {
        $other = User::factory()->create(['role' => 'breeder']);
        $otherProfile = BreederProfile::factory()->create(['user_id' => $other->id]);
        $otherDoc = BreederDocument::factory()->create(['breeder_profile_id' => $otherProfile->id]);

        $this->actingAs($this->breeder, 'sanctum')
            ->postJson('/api/v1/verification-requests', [
                'document_ids' => [$otherDoc->id],
            ])
            ->assertStatus(422)
            ->assertJsonValidationErrors(['document_ids']);
    }

    public function test_rejection_notes_visible_in_status_response(): void
    {
        VerificationRequest::factory()->create([
            'breeder_profile_id' => $this->profile->id,
            'status'             => 'rejected',
            'admin_notes'        => 'Certificate expired.',
        ]);

        $this->actingAs($this->breeder, 'sanctum')
            ->getJson('/api/v1/verification-requests/status')
            ->assertJsonPath('data.status', 'rejected')
            ->assertJsonPath('data.admin_notes', 'Certificate expired.');
    }

    public function test_status_returns_verified_for_verified_breeder(): void
    {
        $this->profile->update(['verified_at' => now()]);

        $this->actingAs($this->breeder, 'sanctum')
            ->getJson('/api/v1/verification-requests/status')
            ->assertJsonPath('data.status', 'verified');
    }

    public function test_buyer_cannot_submit_verification_request(): void
    {
        $buyer = User::factory()->create(['role' => 'buyer']);
        $buyer->assignRole('buyer');

        $this->actingAs($buyer, 'sanctum')
            ->postJson('/api/v1/verification-requests', ['document_ids' => [$this->doc->id]])
            ->assertStatus(403);
    }
}
```

### §3 — Architecture Note: `verification_requests` Uses Direct FK, Not Polymorphic

The architecture schema shows `verification_requests` as polymorphic (`requestable_type`, `requestable_id`) to support both `BreederProfile` and `SitterProfile`. However, for MVP, only `BreederProfile` is implemented. The migration in this story uses a direct `breeder_profile_id` FK for simplicity.

When Epic 5 adds Sitter verification, the table should be migrated to the polymorphic pattern (or a new `sitter_verification_requests` table created — discuss with architect). This is noted as technical debt.

---

### §3b — API Route Discrepancy vs Epic Source

The route registered in T8 is `POST /api/v1/verification-requests` (RESTful, consistent with all other routes in this project). The parent Epic 3.1 AC#1 states `POST /api/verification/submit` — this path predates the `/api/v1/` versioning convention documented in architecture/rest-api-spec.md and is superseded by the story's implementation. All frontend calls and tests use the story's path.
