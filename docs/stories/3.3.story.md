# Story 3.3: Verified/Unverified Badge Display

**Epic:** 3 — Verification System
**Status:** Approved

---

## Story

**As a** buyer browsing listings and breeder profiles,
**I want** to clearly see whether a breeder is verified or unverified,
**so that** I can make an informed decision about who to trust when adopting a pet.

---

## Acceptance Criteria

1. Verified breeders display a green checkmark badge with the label "Verified" on: listing cards (browse page), listing detail page, and breeder profile page.
2. Unverified breeders display a grey "Unverified" label in the same locations — the badge is always shown, never hidden.
3. The badge state is driven exclusively by the `verified` boolean in the API response (derived from `breeder_profiles.verified_at`). No client-side caching of the verified state beyond the current TanStack Query cache lifetime.
4. Each badge includes a tooltip: Verified → "This breeder has been verified by PawTrust"; Unverified → "This breeder has not yet been verified".
5. On mobile viewports (< 640px), the badge must appear above the fold on the listing card and breeder profile header — not hidden below a scroll boundary.
6. The `?verified_only=true` filter on the browse/search page correctly excludes unverified breeders (already implemented in Story 2.4 — this story adds a UI toggle for it and confirms end-to-end correctness).

## Dev Notes

### §1 — Badge Placement Strategy (Mobile-First)

The "above the fold" requirement on mobile means the badge must be placed in the first ~480px of the page when rendered at 375px width. Placement rules:

| Surface         | Required placement                                                                                             |
| --------------- | -------------------------------------------------------------------------------------------------------------- |
| Listing card    | Below photo, above price — within card body top section                                                        |
| Listing detail  | In the breeder summary block in the top-right column (desktop) or below the main photo (mobile stacked layout) |
| Breeder profile | Inside the profile header component, adjacent to the kennel name, before the description section               |

If the current layout puts the kennel name below a fold on mobile, the badge placement task should also move the kennel name up, not just append the badge to the existing position.

### §2 — Data Flow: No Extra API Call Needed

The `verified` boolean is already embedded in:

- `GET /api/v1/breeders/{id}` → `BreederProfileResource` → `verified: bool`
- `GET /api/v1/listings` → `ListingResource` → `breeder.verified: bool`
- `GET /api/v1/listings/{id}` → same

No dedicated "check verification status" call is needed for badge rendering. The badge simply reads from the already-loaded TanStack Query cache for that resource.

Cache invalidation is implicit: when Story 3.2 admin approval writes `verified_at` to the DB, the next time a public page fetches the listing/profile, it will receive `verified: true`. TanStack Query's `staleTime` (set globally in Story 1.4's `queryClient` config) governs how quickly the change propagates to already-open pages. This is acceptable — real-time badge update is not a requirement; page-reload freshness is sufficient.

### §3 — `verified_only` Filter — Backend Confirmation

Story 2.4 stubbed this filter. This story confirms it is fully wired. The correct query is:

```php
// ListingService.php
if ($filters['verified_only'] ?? false) {
    $query->whereHas(
        'breederProfile',
        fn(Builder $q) => $q->whereNotNull('verified_at')
    );
}
```

This does NOT join the `breeder_profiles` table globally — it uses an existence subquery, which is safe for pagination and avoids duplicate rows. Do not rewrite to a `join` without careful consideration of the `DISTINCT` implications.

### §4 — Tooltip Implementation Notes

Use the native HTML `title` attribute for the tooltip text AND a visible `aria-label` for accessibility. A custom styled tooltip (CSS `:hover + popup div`) is out of scope for this story. If the design requires a custom tooltip in future, it should be a separate component update.

```tsx
<span className={badgeClasses} title={tooltipText} aria-label={tooltipText} role="status">
  {icon} {label}
</span>
```
