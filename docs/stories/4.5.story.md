# Story 4.5: Buyer Dashboard

**Epic:** 4 ظ¤ Inquiry System & Contact Reveal
**Status:** Approved

---

## Story

**As a** buyer,
**I want** a dashboard where I can track my inquiries, reviews, and saved listings,
**so that** I can manage all my activity on PawTrust in one place.

---

## Acceptance Criteria

1. BuyerDashboard route: `/dashboard/buyer`. Auth required, buyer role only. This is the landing page after buyer login.
2. **My Inquiries** section: paginated list of all inquiries submitted by the buyer. Each inquiry card shows: listing photo (thumbnail), breed/title, breeder name, submission date, status badge, "View Contact Details" button (always available post-submission), and "View Documents" button (from Story 4.3).
3. **My Reviews** section: list of reviews submitted by the buyer. Each row: breeder kennel name, star rating (display), date posted, first 100 chars of review text.
4. **Saved Listings** section: listings the buyer has bookmarked. Heart icon on listing cards adds/removes from saved list. Endpoints: `POST /api/v1/listings/{id}/save` and `DELETE /api/v1/listings/{id}/save`. Saved listings persist across sessions (stored in DB, not localStorage).
5. Each section has a meaningful empty state with a CTA directing the buyer to browse listings.
6. BuyerDashboard is the landing page for buyers after login (redirect from `/dashboard` based on role).

### ┬د1 ظ¤ Breeder Mark-as-Read Endpoint

Story 4.2 added `read_at` to the `inquiries` table and the unread count endpoint. This story completes the loop: when a _breeder_ views an inquiry's detail, `read_at` should be set. Add a `GET /breeder/inquiries/{inquiry}` detail endpoint to `BreederInquiryController` that:

1. Validates the inquiry belongs to the breeder's listing (not just any inquiry)
2. Sets `inquiry.read_at = now()` if not already set
3. Returns full inquiry detail (complete message, not truncated)

This endpoint was deferred from Story 4.2 to keep that story focused. Implement it as part of this story since the BuyerDashboard story provides the full picture of inquiry read management.

---

### ┬د2 ظ¤ `is_saved` in `ListingResource`

The `is_saved` field on `ListingResource` is computed per-listing. On the browse page (`GET /listings`), this means a query per listing to check if the current user has saved it. For the MVP browse page (20 listings/page), this is 20 extra queries. Mitigate with:

```php
// In ListingController::index(), after paginating:
if (auth()->check() && auth()->user()->hasRole('buyer')) {
    $savedIds = auth()->user()->savedListings()
        ->whereIn('listing_id', $listings->pluck('id'))
        ->pluck('listing_id')
        ->toSet();  // use collect
}
// Pass $savedIds into the ResourceCollection via 'additional' or use a per-request singleton
```

For MVP simplicity, the N+1 approach is acceptable if it degrades gracefully. Add a comment in `ListingResource` noting this as a known performance hotspot for future optimization.

---

### ┬د3 ظ¤ Contact Details Persistence on BuyerDashboard

The inquiry submission response (Story 4.1) includes `breeder_contact`. This data must be available again when the buyer returns to their dashboard. Options:

1. **Store on server** ظ¤ include `breeder_contact` in `GET /buyer/inquiries` response (the contact is already "revealed" and the relationship exists)
2. **Re-derive** ظ¤ the `BuyerInquiryResource` re-fetches the contact from `listing.breederProfile.user` (same data, always available since the inquiry record exists)

Use option 2 ظ¤ no additional state needed. `BuyerInquiryResource` includes `breeder_contact` by eagerly loading `inquiry.listing.breederProfile.user`. This is consistent with the Story 4.1 `InquiryResource` pattern.

**SECURITY REMINDER:** `breeder_contact.email` and `breeder_contact.phone` are returned here because the buyer already has an inquiry. This endpoint is behind `auth:sanctum` + `role:buyer` + the buyer's own inquiries only. It is NOT a public endpoint. The same contact data restriction from Story 4.1 Dev Notes ┬د1 applies.

```bash
LISTING_ID=1

# 1. Save a listing
curl -s -X POST "http://localhost:8000/api/v1/buyer/saved-listings/$LISTING_ID/save" \
  -H "Authorization: Bearer $BUYER_TOKEN" | jq '.data.saved'
# ظْ true

# 2. List saved listings
curl -s "http://localhost:8000/api/v1/buyer/saved-listings" \
  -H "Authorization: Bearer $BUYER_TOKEN" | jq '.data | length'
# ظْ 1 (and is_saved: true in each item)

# 3. Unsave
curl -s -X DELETE "http://localhost:8000/api/v1/buyer/saved-listings/$LISTING_ID/save" \
  -H "Authorization: Bearer $BUYER_TOKEN" | jq '.data.saved'
# ظْ false

# 4. Buyer inquiries list
curl -s "http://localhost:8000/api/v1/buyer/inquiries" \
  -H "Authorization: Bearer $BUYER_TOKEN" | jq '.data[0] | {listing: .listing.title, has_reviewed}'

# 5. Buyer reviews list
curl -s "http://localhost:8000/api/v1/buyer/reviews" \
  -H "Authorization: Bearer $BUYER_TOKEN" | jq '.data | length'

# 6. Login and confirm redirect to /dashboard/buyer
# ظْ Open http://localhost:5173/login, sign in as buyer ظْ should land on /dashboard/buyer
```
