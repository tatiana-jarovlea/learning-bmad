# Story 4.2: Breeder Notification on New Inquiry

**Epic:** 4 — Inquiry System & Contact Reveal
**Status:** Draft

---

## Story

**As a** breeder,
**I want** to be notified when a buyer submits an inquiry,
**so that** I can respond promptly and manage incoming interest in my listings.

---

## Acceptance Criteria

1. On inquiry submission (Story 4.1): email sent to breeder via Laravel Queue → AWS SES. Email must be queued (async) — no synchronous sending in the controller.
2. Email contains: buyer's name, the optional message (or "No message provided" if empty), listing title, and a direct link to the BreederDashboard inquiry list.
3. Email is rendered in the breeder's locale (`users.locale` — `ro` or `ru`).
4. BreederDashboard navigation shows a badge/counter with the number of **unread** inquiries. The count updates when the page is loaded; real-time push is not required at MVP.
5. BreederDashboard inquiry list (`/dashboard/breeder/inquiries`) shows all inquiries for the breeder's listings. Each row: buyer name, date submitted, listing title, message excerpt (first 100 chars), status.
6. Can be verified locally: submit inquiry from a buyer account, check `storage/logs/laravel.log` for the breeder notification email.

### §1 — Buyer Contact Details: Breeder Does NOT Receive Them

The inquiry system is one-directional at MVP: buyer contacts breeder. The breeder receives the buyer's name and message but NOT the buyer's email or phone number. The breeder is expected to respond via the listing's listed contact method (which the buyer already has after submitting the inquiry).

This is intentional — adding buyer contact details to the breeder notification email would bypass the platform contact model and is a post-MVP decision requiring a privacy policy review.

---

### §2 — Unread Count Polling Strategy

The unread count badge uses TanStack Query polling: `staleTime: 60_000` (1 minute). On focus, the query refetches automatically (default TanStack Query behavior with `refetchOnWindowFocus: true`). A breeder switching tabs and returning to the dashboard will see a fresh count within 1 minute or immediately on focus.

WebSocket / SSE real-time push for the badge is explicitly out of scope for MVP. If the product team requests real-time notification, it should be a separate story.

---

### §3 — `read_at` vs a Separate `notifications` Table

The MVP implementation uses a `read_at` nullable timestamp directly on the `inquiries` table. This is simpler than a generic `notifications` table (which would require polymorphic lookups and a notification type enum).

When a general notification center is added post-MVP, the `notifications` table can be introduced and `read_at` on `inquiries` can be deprecated. For now, keep it simple.
