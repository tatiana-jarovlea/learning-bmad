# Story 6.2: GDPR Compliance

**Epic:** 6 — i18n, Polish & Launch Readiness
**Status:** Draft

---

## Story

**As a** user,
**I want** to know how my data is used and have control over it,
**so that** I can use PawTrust with confidence in my privacy rights.

---

## Acceptance Criteria

1. Privacy Policy page (`/privacy`) — available in Romanian and Russian. Covers: data collected, how it's used, third-party processors (AWS, Stripe, SES), user rights (access, rectification, erasure, portability), and a contact address for data requests.
2. Cookie consent banner shown on first visit: "Accept all" / "Reject non-essential" buttons. At MVP, only essential cookies are used (no third-party tracking). Consent decision stored in `localStorage`.
3. Account deletion: `DELETE /api/v1/account` — soft-deletes user (sets `deleted_at`), anonymizes PII (name → `"Deleted User"`, email → `"deleted_{id}@pawtrust.md"`), revokes all Sanctum tokens, queues S3 document deletion (completed within 30 days). Returns HTTP 200.
4. Data export: `GET /api/v1/account/export` — returns JSON of all user-owned data (profile, listings, inquiries, reviews, saved listings). Intended for manual fulfillment of data subject access requests (DSAR).
5. Terms of Service page (`/terms`) — linked from footer, available in both languages.
6. All registration and contact forms include a privacy notice: "By registering, you agree to our [Privacy Policy] and [Terms of Service]."
7. Footer links: Privacy Policy, Terms of Service, Contact Us (email address shown as text — no mailto link to avoid scraping).

  ```php
  public function deleteAccount(User $user): void
  {
      DB::transaction(function () use ($user) {
          // 1. Anonymize PII in place (required even for soft-deleted records
          //    in case of partial DB restore or GDPR audit)
          $user->forceFill([
              'name'  => 'Deleted User',
              'email' => "deleted_{$user->id}@pawtrust.md",
              'phone' => null,
          ])->save();

          // 2. Revoke all Sanctum tokens
          $user->tokens()->delete();

          // 3. Soft-delete the user record
          $user->delete();
      });

      // 4. Queue S3 document cleanup (outside transaction — async)
      DeleteUserS3DocumentsJob::dispatch($user->id)->delay(now()->addMinutes(5));
  }
  ```

  - Anonymization happens BEFORE soft-delete so that the anonymized data is what is soft-deleted (audit log remains consistent)

## Dev Notes

### §1 — Soft Delete Cascade Considerations

When a user is soft-deleted, their related records (listings, inquiries, reviews, breeder profile) are NOT automatically soft-deleted — Eloquent SoftDeletes only applies to the model it is added to. This is intentional:

- **Breeder listings**: remain visible to buyers who have open inquiries, but will be excluded from browse results because the breeder profile query filters on `users.deleted_at IS NULL`. Verify `ListingService::list()` joins through to the user and respects soft delete.
- **Inquiries**: buyer's inquiries remain for breeder's records (the breeder still received the contact). The anonymized buyer name `"Deleted User"` is acceptable here.
- **Reviews**: remain published under `"Deleted User"` name — this is standard practice (review persists, author anonymized).
- **Saved listings**: can remain in DB — they become orphaned but cause no harm; the FK cascade on `buyer_id` will clean them up anyway since `saved_listings` has a `cascadeOnDelete` constraint (Story 4.5 T1).

Add a comment in `AccountDeletionService` explaining this deliberate design.

---

### §2 — DSAR (Data Subject Access Request) Export Scope

The `GET /api/v1/account/export` endpoint fulfills GDPR Article 20 (data portability). At MVP, the export is returned as a JSON response that the frontend triggers as a file download. It does NOT include:

- S3 file binaries (documents/photos) — the export lists them by filename only
- Audit log entries (internal administrative data — not subject to portability)
- IP addresses stored in `sitter_contact_submissions` (these are operational security data, not personal data uploaded by the user)

This scope is defensible for MVP. For full GDPR compliance, a DPO review before launch is recommended.

---

### §3 — Account Deletion & Active Listings

When a breeder deletes their account:

- Their active listings should be set to `status = 'deleted'` (soft-delete or status change) so they no longer appear in search
- The `DeleteUserS3DocumentsJob` handles S3 photo/document cleanup
- The `breeder_profiles` soft-delete is not implemented in this story — the anonymization of the user record is sufficient at MVP. The breeder profile remains in the DB but is effectively inaccessible because the user it belongs to is deleted.

A stricter implementation (cascading soft-deletes to breeder profile and listings) can be added post-MVP. Add a `// TODO: cascade soft-delete to breeder_profiles and listings on user deletion` comment in `AccountDeletionService`.

### §5 — What This Story Does NOT Include

- Automated S3 deletion within strict 30-day SLA (the job is dispatched immediately but retried if failed — SLA tracking is post-MVP)
- Consent Management Platform (CMP) with granular cookie categories (no third-party tracking at MVP)
- Right to rectification UI (user can update their profile — that is rectification — no separate endpoint needed)
- GDPR Data Processing Agreement (DPA) with AWS, Stripe (legal → not a code task)
- DPO appointment notice in Privacy Policy (legal → not a code task)
- Breach notification workflow (post-MVP operational process)

```bash
# 3. Confirm login fails
curl -s -X POST http://localhost:8000/api/v1/auth/login \
  -H "Content-Type: application/json" \

# mysql: SELECT name, email FROM users WHERE id = $USER_ID;
```
