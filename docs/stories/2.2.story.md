# Story 2.2: Pet Listing Creation with Photos

**Epic:** 2 — Breeder Profiles & Listings
**Status:** Draft

---

## Story

**As a** breeder,
**I want** to create a pet listing with photos and a health document,
**so that** buyers can discover my available pets and assess them before making an inquiry.

---

## Acceptance Criteria

1. Listing creation form collects: species (dog/cat), breed (searchable dropdown filtered by species), gender, date of birth, price (EUR), location (inherited from breeder profile, overridable), title, description (max 500 chars), listing type (standard €3 / featured €8).
2. Minimum 3 listing photos required; minimum 1 parent photo required. All uploaded to private S3 bucket. Photos served via CloudFront pre-signed URLs (24h TTL).
3. Photo upload: max 5MB per photo, JPEG/PNG/WebP only. Max 10 photos per listing total. File type validated server-side by checking magic bytes (not just MIME type / extension).
4. Health certificate upload required (PDF or image, max 10MB). Stored on S3, private. Pre-inquiry: shown as an icon label only — no URL returned.
5. Listing saved with status `draft` until payment completes. In Epic 2 dev/test: `SKIP_LISTING_PAYMENT=true` in `.env` bypasses payment and sets status to `active` immediately.
6. With bypass enabled: `POST /api/v1/listings` saves and returns listing with `status: "active"`.
7. `GET /api/v1/listings/{id}` returns full listing data. Health certificate shown as `{ "health_certificate": "on_file" }` — no URL returned.
8. Breeder dashboard lists all their own listings with status (active/draft/expired/sold).

    ```php
    private const ALLOWED_SIGNATURES = [
        'jpg'  => "\xFF\xD8\xFF",        // JPEG
        'png'  => "\x89\x50\x4E\x47",   // PNG
        'webp' => "RIFF",                // WebP (bytes 0-3 are RIFF, bytes 8-11 are WEBP)
    ];

    public function validateMagicBytes(UploadedFile $file): void
    {
        $handle = fopen($file->getRealPath(), 'rb');
        $header = fread($handle, 12);
        fclose($handle);

        $isJpeg  = str_starts_with($header, self::ALLOWED_SIGNATURES['jpg']);
        $isPng   = str_starts_with($header, self::ALLOWED_SIGNATURES['png']);
        $isWebp  = str_starts_with($header, 'RIFF') && substr($header, 8, 4) === 'WEBP';

        if (!$isJpeg && !$isPng && !$isWebp) {
            throw ValidationException::withMessages([
                'photo' => 'File must be a valid JPEG, PNG, or WebP image.',
            ]);
        }
    }
    ```

## Dev Notes

### §1 — Listing Status Lifecycle

```
draft → [payment completed / SKIP_LISTING_PAYMENT=true] → active
active → [breeder marks sold] → sold
active → [expires after listing period] → expired
```

In Epic 2, only the `draft → active` transition is implemented (either via Stripe webhook in Epic 5, or via `SKIP_LISTING_PAYMENT` bypass). The breach cases (`sold`, `expired`) are handled in later stories.

---

### §2 — Two-Step Upload Flow

Listings are created in a multi-step process:

1. `POST /api/v1/listings` — create listing with text data → returns `{ id, status: 'draft' }`
2. `POST /api/v1/listings/{id}/photos` — upload each photo (called per file)
3. `POST /api/v1/listings/{id}/health-certificate` — upload health doc
4. If `SKIP_LISTING_PAYMENT=true`: listing becomes `active` at step 1. Otherwise stays `draft` until Stripe webhook fires.

This avoids bundling large file uploads into a single request and gives finer error reporting per file. The frontend multi-step form aligns with this sequence.

---

### §3 — Magic Bytes Validation (Security)

MIME type validation (`mimes:jpg,jpeg,png,webp`) is bypassed by renaming a malicious file (e.g. `shell.php` → `photo.jpg`). Magic bytes validation reads the actual binary header of the file, which cannot be falsified without corrupting the file.

JPEG: starts with `\xFF\xD8\xFF`
PNG: starts with `\x89PNG`
WebP: starts with `RIFF....WEBP`
PDF: starts with `%PDF`

The `validateMagicBytes` method in `ListingPhotoService` must be called BEFORE streaming to S3. If validation fails, the file is rejected and never stored. [Source: architecture/security.md — OWASP A08: Software and Data Integrity]
