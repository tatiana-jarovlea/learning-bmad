# Story 1.4: Role-Based Dashboard Shells

**Epic:** 1 — Foundation & Authentication
**Status:** Draft

---

## Story

**As a** logged-in user,
**I want** to land on a dashboard tailored to my role immediately after login,
**so that** I have an appropriate starting point with relevant navigation and I understand what I can do on the platform.

---

## Acceptance Criteria

1. Three distinct dashboard shell pages exist: `BuyerDashboard`, `BreederDashboard`, `AdminDashboard`. `SitterDashboard` exists as a stub only (full implementation deferred to Epic 5).
2. After a successful login, the frontend reads `user.role` from the auth store and navigates to the correct dashboard route (`/buyer/dashboard`, `/breeder/dashboard`, `/admin`, `/sitter/dashboard`).
3. Each dashboard shell renders:
   - A shared `Header` with: Logo, Browse Listings, Pet Sitters, language switcher (RO/RU), and a user dropdown menu.
   - A role-specific `Sidebar` with empty-state-ready navigation links (desktop) or a tab bar (mobile).
   - A main content area showing an empty-state placeholder that is role-appropriate.
4. User dropdown menu contains: Dashboard link, Profile (stub link), Logout button. Clicking Logout calls the store's `logout()` action from Story 1.3, then redirects to `/`.
5. `ProtectedRoute` wrapper: any route inside it that is accessed without a valid token redirects to `/login`.
6. `AdminRoute` wrapper: `/admin` route redirects non-admin authenticated users to `/` (their own dashboard). Unauthenticated users are redirected to `/login`.
7. Language switcher (`LocaleSwitcher`) toggles between RO and RU, persists the choice to `localStorage`, and applies it application-wide via react-i18next.
8. Navigation labels and empty-state copy are translated — both RO and RU translation keys populated for all strings introduced in this story.

## Dev Notes

### §1 — Route Guard Pattern

`ProtectedRoute` and `AdminRoute` are React Router v6 layout routes — they render `<Outlet />` when passing, or `<Navigate />` when blocking. No HOC pattern needed.

```tsx
// Usage in router.tsx
{
  element: <ProtectedRoute />,   // acts as a guard wrapper
  children: [
    { path: '/buyer/dashboard', element: <BuyerDashboard /> },
  ],
}
```

Do NOT use the v5 pattern of wrapping `<Route element={<PrivateRoute><Component /></PrivateRoute>}>` — that is outdated.

---

### §2 — DashboardLayout vs Page-Level Layout

`DashboardLayout` is a React component, not a router layout route. Each dashboard page imports and uses it:

```tsx
// BuyerDashboard.tsx
const BuyerDashboard = () => (
  <DashboardLayout>
    <main>...</main>
  </DashboardLayout>
)
```

This is preferred over a router-level layout for dashboards because each dashboard has slightly different sidebar content based on role, and `DashboardLayout` reads the role from the auth store internally. A single router-level layout cannot be easily parameterized per role in React Router v6 without additional context passing.

---

### §3 — Logout from UserMenu

`logout()` in the auth store is async (calls `POST /api/v1/auth/logout` then clears local state). The `UserMenu` component:

```tsx
const handleLogout = async () => {
  setIsLoggingOut(true)
  await useAuthStore.getState().logout()
  navigate('/')
}
```

The `finally` block in `logout()` (Story 1.3, T12) ensures local state always clears. Even if the API call fails (network error, already expired token), the user is logged out locally.
