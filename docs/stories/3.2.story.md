# Story 3.2: Admin Verification Review Dashboard

**Epic:** 3 — Verification System
**Status:** Draft

---

## Story

**As an** admin,
**I want** to review breeder verification submissions and approve or reject them,
**so that** I can maintain platform trust and issue the Verified badge to legitimate breeders.

---

## Acceptance Criteria

1. Admin dashboard route `/admin/verification` lists all verification submissions with status filter (pending, under_review, approved, rejected). Paginated (20 per page).
2. Each submission row shows: breeder name, kennel name, submission date, document count with view links, current status.
3. Admin can view individual documents via 15-minute pre-signed S3 URLs. Endpoint: `GET /api/v1/admin/documents/{id}/preview`. Admin role required.
4. Admin can: set status to `under_review` (locks resubmission), `approved` (triggers Verified badge + `verified_at` timestamp), `rejected` (requires notes, min 20 chars).
5. Status update endpoint: `PUT /api/v1/admin/verification-requests/{id}/review` — body: `{ status, notes }`. Admin role required.
6. On approval: breeder's `verified_at` set to `now()`. Reflected immediately on public profile and all listings.
7. On rejection: breeder's `verified_at` remains `null`. Rejection notes stored; visible to breeder via Story 3.1 status endpoint.
8. All admin review actions logged in an `admin_audit_logs` table with: admin user ID, action, target type/ID, timestamp.

## Dev Notes

### §1 — Approval Atomicity

When `status === 'approved'`:

1. `verification_requests.status` → `'approved'`
2. `breeder_profiles.verified_at` → `now()`

Both updates happen in the same `AdminVerificationService::review()` call. They should be wrapped in a DB transaction to prevent partial updates:

```php
DB::transaction(function () use ($vr, $admin, $status, $notes) {
    $vr->update([...]);
    if ($status === 'approved') {
        $vr->breederProfile->update(['verified_at' => now()]);
    }
});
app(AdminAuditService::class)->log(...);  // outside transaction — audit log failure should not roll back the decision
```

The audit log is written outside the transaction deliberately. A failed audit write should NOT roll back the approval — the core business operation (granting verification) takes priority. The missing audit log entry is a recoverable issue; an unverified state for a breeder who passed review is not. [Source: architecture/security.md]

### §3 — Document Preview URL: On-Demand Only

`GET /admin/documents/{id}/preview` generates a new pre-signed URL on every request. The URL is NOT cached server-side. With a 15-minute TTL and single-use intent (admin opens the document in their browser), this is the correct
pattern.

The frontend calls this endpoint when the admin clicks "Preview" — not on page load. This avoids generating URLs for all documents upfront, which would burn pre-sig URL quota unnecessarily and return URLs that expire before the admin reviews them.

---

### §3b — HTTP Method Override: PUT (not POST)

The Epic 3 specification lists the review endpoint as `POST /api/admin/verification/{id}/review`. This story implements it as `PUT /api/v1/admin/verification-requests/{id}/review`.

`PUT` is semantically correct — the request sets (overwrites) the review decision on a specific resource and is idempotent: submitting the same `status` + `notes` twice produces the same state. `POST` in the Epic overview was shorthand. The canonical endpoint is `PUT /api/v1/admin/verification-requests/{id}/review`.
