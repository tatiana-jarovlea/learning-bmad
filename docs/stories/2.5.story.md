# Story 2.5: Listing Detail Page

**Epic:** 2 — Breeder Profiles & Listings
**Status:** Draft

---

## Story

**As a** buyer,
**I want** to view a full listing page with photos, breeder context, and document indicators,
**so that** I can thoroughly assess a pet and feel confident before submitting an inquiry.

---

## Acceptance Criteria

1. `GET /api/v1/listings/{id}` returns full listing data: all listing fields, all photos (CloudFront pre-signed URLs, 1-hour TTL), parent photos, breeder profile summary (kennel name, location, verified status, years active, breed specializations), cynology achievement highlights (count + up to 3 titles), review summary (average rating + count).
2. Health certificate shown as `"health_certificate": "on_file"` — no S3 key or download URL in response (pre-inquiry).
3. Contact details (phone, email) NOT in response under any key.
4. Frontend: "Submit Inquiry to Access Contact & Documents" CTA button visible to authenticated buyers; prompts unauthenticated users to log in (login redirect preserves the listing URL as `?redirect=/listings/{id}`).
5. Cynology achievements section: renders list of `{ title, federation, year }` records from breeder profile (0–∞ records; empty state shown gracefully).
6. Reviews section: last 5 published reviews (star rating + text excerpt), average rating, total count. "See all reviews" links to `/breeders/{id}#reviews`. No reviews yet = empty state.
7. Breadcrumb: Home → Browse Pets → [Listing Title].
8. Can be verified locally: create listing → fetch `GET /listings/{id}` → confirm no phone/email/s3 key in response.

  - File: `app/Http/Controllers/Api/V1/ListingController.php` (update from Story 2.2)

    ```php
    public function show(int $id): JsonResponse
    {
        $listing = Listing::with([
            'photos',
            'breederProfile',
            'breederProfile.achievements' => fn($q) => $q->latest()->limit(3), // max 3 highlights on detail page
            'reviews' => fn($q) => $q->published()->with('buyer')->latest()->limit(5),
        ])
        ->where('status', 'active')
        ->findOrFail($id);

        // Separate aggregation query — eager-load limit(5) on reviews cannot compute full avg/count
        $reviewSummary = Review::where('listing_id', $listing->id)
            ->published()
            ->selectRaw('AVG(rating) as average, COUNT(*) as total')
            ->first();

        return response()->json([
            'data' => ListingResource::make($listing)
                ->additional([
                    'review_summary' => [
                        'average_rating' => $reviewSummary?->average ? round($reviewSummary->average, 1) : null,
                        'total_count'    => (int) ($reviewSummary?->total ?? 0),
                    ],
                ]),
        ]);
    }
    ```

  - Returns 404 for draft/sold/expired listings — buyers cannot access non-active listings
  - Review summary (average + count) computed via extra query against all published reviews (not just the last 5):
    ```php
    $reviewSummary = Review::where('listing_id', $listing->id)
        ->published()
        ->selectRaw('AVG(rating) as average, COUNT(*) as total')
        ->first();
    ```
    Pass into `ListingResource` via `->additional(['review_summary' => [...]])`. In the resource, access via `$this->additional['review_summary']`. The `review_summary` key will appear at the top level of the `data` object in the response (alongside the listing fields).
    > **Why `additional()`?** The summary is computed separately from the listing model — it doesn't belong on the Eloquent model itself. `additional()` on a `JsonResource` merges extra keys into the serialized response without polluting the model.

### §1 — Photo URL TTL: 1 Hour for Detail, 24 Hours for Cards

| Context                                    | TTL    | Reason                                            |
| ------------------------------------------ | ------ | ------------------------------------------------- |
| Search result card (`ListingCardResource`) | 24h    | High volume, CDN-cacheable                        |
| Listing detail (`ListingResource`)         | 1 hour | More sensitive context; shorter window acceptable |
| Documents (admin access)                   | 15 min | Maximum sensitivity                               |
| Contact docs (post-inquiry)                | 30 min | Buyer access only after acceptance                |

URLs are generated on each API request — they are not stored. CDN caching means the actual S3 presigned URL is only generated once per cache TTL at the CDN edge. [Source: architecture §7.1]

### §3 — Open Redirect Prevention in Login `?redirect`

`Login.tsx` T21 must validate the `redirect` param before using it:

```typescript
const getRedirectPath = (redirectParam: string | null): string => {
  if (!redirectParam) return null
  // Only allow relative paths — prevent open redirect to external URLs
  try {
    const url = new URL(redirectParam, window.location.origin)
    if (url.origin !== window.location.origin) return null // external URL blocked
    return url.pathname + url.search
  } catch {
    return null
  }
}
```

This follows OWASP A01 — broken access control (open redirect variant). An attacker could craft `?redirect=https://evil.com` to redirect users to a phishing site after login. [Source: architecture/security.md]

### §4 — Review Summary: N+1 Avoidance

The `ListingController::show()` loads the last 5 reviews via eager-load (`limit(5)`). The review summary (average + count over ALL reviews) cannot be done in the same eager-load because `limit()` on an eager-loaded relationship counts only the loaded subset.

Two approaches:

1. **Separate query** (simplest): `Review::where('listing_id', $id)->published()->selectRaw('AVG(rating) as average, COUNT(*) as total')->first()` — one extra query per detail request. Acceptable at MVP traffic.
2. **Denormalized counter** (future optimization): add `review_count` and `review_avg` columns to `listings` table, updated via observer. Out of scope for MVP.

Use approach 1 now. Document approach 2 as a performance optimization story.

### §5 — Reviews: Breeder-Level Query

Reviews are attached to `listings`, not directly to `breeder_profile`. To fetch all reviews for a breeder (for the public breeder profile page reviews section, Story 4.4 and AC6's "See all reviews" link), use:

```php
Review::whereHas('listing', fn($q) => $q->where('breeder_profile_id', $profileId))
    ->published()
    ->with('buyer')
    ->latest()
    ->paginate(10);
```

A direct `breeder_profile_id` column on `reviews` would improve query performance (avoids the join) but adds denormalization. Acceptable as a post-MVP optimization if the join becomes a measurable bottleneck. For MVP, this single-join query on an indexed column is sufficient.
