# Story 6.3: Performance & SEO Foundations

**Epic:** 6 — i18n, Polish & Launch Readiness
**Status:** Draft

---

## Story

**As** PawTrust,
**I want** the platform to load fast and be discoverable on search engines,
**so that** we can grow organically from Moldova/Romania search traffic.

---

## Acceptance Criteria

1. All listing and breeder profile pages have unique `<title>` tags, `<meta name="description">`, and Open Graph tags (`og:title`, `og:description`, `og:image`) populated with actual listing/breeder data.
2. Homepage and key landing pages have JSON-LD structured data for Organization schema.
3. React app uses React Helmet (or `react-helmet-async`) for dynamic `<head>` management.
4. Images: listing and portfolio photos are served as WebP via CloudFront. Lazy loading on all listing grid images. The main listing photo (LCP candidate) is eagerly loaded (`loading="eager"`, `fetchpriority="high"`).
5. Lighthouse audit run on listing detail and search results pages: Performance ≥ 80, LCP < 3s on simulated 4G mobile. Results documented before launch sign-off.
6. `sitemap.xml` generated and available at `/sitemap.xml` — includes all static routes and a script to list public listing/breeder/sitter URLs. Submitted to Google Search Console.
7. `robots.txt` at `/robots.txt` — allows all crawlers on public pages, disallows `/api/`, `/admin/`, `/dashboard/`, `/sitter/` (dashboard), `/buyer/`, `/breeder/`.

<?php

namespace Tests\Feature;

use Tests\TestCase;
use Illuminate\Foundation\Testing\RefreshDatabase;
use App\Models\Listing;
use App\Models\BreederProfile;

class SitemapTest extends TestCase

    {
        $response = $this->get('/sitemap.xml');

        $response->assertStatus(200);
        $response->assertHeader('Content-Type', 'application/xml');
        $this->assertStringStartsWith('<?xml', trim($response->getContent()));
    }

    public function test_sitemap_includes_active_listing_url(): void
    {
        $breeder = User::factory()->create(['role' => 'breeder']);
        $profile = BreederProfile::factory()->create(['user_id' => $breeder->id]);
        $listing = Listing::factory()->create([
            'breeder_profile_id' => $profile->id,
            'status'             => 'active',
        ]);

        $content = $this->get('/sitemap.xml')->getContent();

        $this->assertStringContainsString('/listings/' . $listing->id, $content);
    }

    public function test_sitemap_excludes_inactive_listings(): void
    {
        $breeder = User::factory()->create(['role' => 'breeder']);
        $profile = BreederProfile::factory()->create(['user_id' => $breeder->id]);
        $listing = Listing::factory()->create([
            'breeder_profile_id' => $profile->id,
            'status'             => 'draft',
        ]);

        $content = $this->get('/sitemap.xml')->getContent();

        $this->assertStringNotContainsString('/listings/' . $listing->id, $content);
    }

        $this->assertStringContainsString('Disallow: /api/', $content);
        $this->assertStringContainsString('Disallow: /admin/', $content);
    }
}

### §1 — WebP Conversion Trade-offs

Converting at upload time (Intervention Image) vs CloudFront image optimization:

- **Upload-time conversion (chosen approach)**: simpler, no additional AWS configuration, deterministic — but original format is lost. Acceptable for MVP since photos are user-uploaded content without archival requirements.
- **CloudFront image optimization**: requires AWS CloudFront Functions or Lambda@Edge — more complex, more powerful. Defer to post-MVP if needed.

If Intervention Image is used, the original file is NOT stored — only the WebP version. This is acceptable for listing/portfolio photos. For `BreederDocument` uploads (PDF/DOC medical certificates in Story 2.3), do NOT apply WebP conversion — documents are not images.

---

### §2 — Sitemap Cache Invalidation

The sitemap is cached for 1 hour. When a new listing is published (webhook activates it), the sitemap cache will not include it for up to 1 hour. This is acceptable — Google crawls sitemaps infrequently and listing URL freshness in the sitemap is not critical for SEO at launch scale. If needed, add `Cache::forget('sitemap')` to `ListingPaymentService::handleSuccessfulPayment()`.

---

### §3 — React SPA and SEO

The React SPA does not support server-side rendering (SSR) at MVP. This means search engine crawlers relying on JavaScript execution are needed to index dynamic content. Google's crawler executes JavaScript, so listing detail pages will be indexed — but with some delay. For the Moldova/Romania market at launch scale, this is acceptable.

If organic SEO becomes critical post-MVP, consider migrating to Next.js (SSR/SSG) or adding a prerendering service (prerender.io). Document this in the PRD as a known limitation.

### §5 — What This Story Does NOT Include

- Server-side rendering (SSR) or static site generation (SSG) — post-MVP
- Automatic sitemap regeneration on content changes (post-MVP — cache invalidation hook)
- Google Search Console setup (operational step — documented in deploy checklist)
- Structured data for listings as Product/ItemList schema (post-MVP — useful for rich results)
- Image CDN with on-the-fly resizing via CloudFront image optimization (post-MVP)
- Core Web Vitals monitoring / analytics (post-MVP — Sentry or Datadog)
