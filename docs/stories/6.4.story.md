# Story 6.4: Security Hardening

**Epic:** 6 — i18n, Polish & Launch Readiness
**Status:** Approved

---

## Story

**As** PawTrust,
**I want** the platform hardened against common attacks,
**so that** user data and the platform are protected at launch.

---

## Acceptance Criteria

1. All API endpoints validated for authentication — no unintentionally public endpoints.
2. SQL injection prevention: all queries use Eloquent ORM or parameterized Query Builder. Raw SQL audited (grep for `DB::statement`, `DB::unprepared`, `whereRaw`).
3. XSS prevention: all user-generated content HTML-escaped on output. React JSX auto-escapes strings. Server-side: any HTML rendering validated.
4. CSRF protection: Laravel CSRF middleware active on all non-API (web) routes. API routes use Sanctum token auth (stateless).
5. HTTPS enforced: HTTP requests redirected to HTTPS at load balancer level. `APP_URL` set to `https://`.
6. Security headers on all responses: `X-Content-Type-Options: nosniff`, `X-Frame-Options: DENY`, `Strict-Transport-Security` (HSTS, 1 year), `Content-Security-Policy` (baseline), `Referrer-Policy: strict-origin-when-cross-origin`.
7. S3 bucket policy audit: no public read access on document/photo buckets. Only pre-signed URL or CloudFront signed access allowed.
8. Stripe webhook signature verification confirmed active in production config — no env variable override that disables it.
9. Rate limiting verified on all three critical endpoints: login (10/min/IP), inquiry submission (5/hour/IP), sitter contact form (5/hour/IP).
10. Dependency audit: `composer audit` and `npm audit` run. No unaddressed critical or high CVEs at launch.

- [ ] **T2** — Audit raw SQL usage
  - Run: `grep -rn "DB::statement\|DB::unprepared\|whereRaw\|selectRaw\|havingRaw\|orderByRaw" app/`

    - Verify no user-controlled input is interpolated directly (must use bindings array)
    - If user input present: rewrite to use parameterized binding `DB::statement('...', [$userInput])`

- [ ] **T3** — Add security headers middleware
  - File: `app/Http/Middleware/SecurityHeaders.php`

  ```php
  class SecurityHeaders
  {
      public function handle(Request $request, Closure $next): mixed
      {
          $response = $next($request);

          $response->headers->set('X-Content-Type-Options', 'nosniff');
          $response->headers->set('X-Frame-Options', 'DENY');
          $response->headers->set('Referrer-Policy', 'strict-origin-when-cross-origin');
          $response->headers->set(
              'Strict-Transport-Security',
              'max-age=31536000; includeSubDomains'
          );
          $response->headers->set(
              'Content-Security-Policy',
              "default-src 'self'; " .
              "script-src 'self' 'unsafe-inline' https://js.stripe.com; " .
              "frame-src https://js.stripe.com https://hooks.stripe.com; " .
              "img-src 'self' data: https://*.cloudfront.net; " .
              "connect-src 'self' https://api.stripe.com; " .
              "style-src 'self' 'unsafe-inline';"
          );

          return $response;
      }
  }
  ```

  - Register in global middleware (applied to all responses including API)
  - **Note on CSP:** `'unsafe-inline'` for scripts is a CSP weakness — required for Stripe.js at MVP. Post-MVP: move to a nonce-based CSP once Stripe SDK version supports it.
  - Stripe domains must be in `frame-src` and `connect-src` for Stripe Checkout and Elements to function

### §1 — CSP and Stripe

The Content Security Policy allows `https://js.stripe.com` in `script-src` and `frame-src`. This is required for Stripe Checkout redirect and Stripe Elements (if used in future). The `'unsafe-inline'` in `script-src` is a weakness — it is included because React + Vite generates inline scripts for module loading in some configurations.

Post-MVP action: switch to a Vite build with a nonce-based CSP to eliminate `'unsafe-inline'`. This requires nonce injection in the HTML template, which is non-trivial with a pure static SPA.

---

### §2 — Security Headers on API vs Web Routes

Security headers should be applied to all responses — both API JSON responses and web routes. The `SecurityHeaders` middleware registered globally covers both. The only exception is the Stripe webhook endpoint — its response is consumed by Stripe servers, not browsers, so the headers are irrelevant there but harmless to include.

---

### §3 — Open Redirect Guard (Reminder)

Story 1.3 implemented an open redirect guard on the login `?redirect` param. Confirm this guard is still in place:

- Only allow redirects to paths starting with `/` (relative paths only)
- Reject any `redirect` param that starts with `//`, `http://`, or `https://`

This is a critical security control — a bypass here allows phishing via the platform's login page.
