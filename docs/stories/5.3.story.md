# Story 5.3: Stripe Pay-per-Listing Integration

**Epic:** 5 — Pet Sitter Profiles & Pay-per-Listing
**Status:** Draft

---

## Story

**As a** breeder,
**I want** to pay for my listing via Stripe,
**so that** my listing goes live and is visible to buyers.

---

## Acceptance Criteria

1. When a listing is created with `SKIP_LISTING_PAYMENT=false` (production mode), the listing is saved with status `pending_payment` and is NOT visible in search results.
2. Payment initiation: `POST /api/v1/listings/{id}/payment/initiate` returns a Stripe Checkout Session URL. Auth required, breeder role, breeder must own the listing.
3. Stripe Checkout configured with two products: standard listing €3.00, featured listing €8.00. PCI-compliant hosted Checkout page — no raw card data touches PawTrust servers.
4. Webhook endpoint: `POST /api/v1/webhooks/stripe` — handles `checkout.session.completed` event. On success: sets listing `status = active`, sets `listed_at = now()`. Webhook signature verified using `STRIPE_WEBHOOK_SECRET`. Webhook is excluded from CSRF and global auth middleware.
5. On payment success: listing immediately appears in search results.
6. On payment failure or abandonment: listing remains `pending_payment`. Breeder can retry payment from BreederDashboard.
7. BreederDashboard listing manager shows listing status and "Complete Payment" CTA for `pending_payment` listings.
8. Stripe test keys (`STRIPE_KEY`, `STRIPE_SECRET`) used in local/staging. Production keys in production.
9. Verifiable locally: use Stripe CLI + test card, complete checkout, verify webhook fires, listing status changes to `active`.

    ```php
    public function handle(Request $request): JsonResponse
    {
        $payload   = $request->getContent();
        $signature = $request->header('Stripe-Signature');

        try {
            $event = $this->stripeService->constructWebhookEvent($payload, $signature);
        } catch (\Stripe\Exception\SignatureVerificationException $e) {
            return response()->json(['error' => 'Invalid signature'], 400);
        }

        if ($event->type === 'checkout.session.completed') {
            $session   = $event->data->object;
            $listingId = $session->metadata->listing_id ?? null;

            if ($listingId) {
                $this->listingPaymentService->handleSuccessfulPayment(
                    (int) $listingId,
                    $session->id
                );
            }
        }

        return response()->json(['received' => true]);
    }
    ```

  - Return **400** for invalid signature (tells Stripe not to retry — the request was rejected). Return **200** for all valid events, including event types we don't handle — Stripe retries on non-2xx responses for handled event types.
  - Route: `POST /api/v1/webhooks/stripe` — excluded from auth and CSRF middleware

- [ ] **T8** — Create `ListingPaymentService`
  - File: `app/Services/ListingPaymentService.php`
  - Method: `handleSuccessfulPayment(int $listingId, string $sessionId): void`

  ```php
  public function handleSuccessfulPayment(int $listingId, string $sessionId): void
  {
      $listing = Listing::find($listingId);

      if (!$listing) {
          Log::warning("Stripe webhook: listing {$listingId} not found");
          return;
      }

      // Idempotency: if already processed this session, skip
      if ($listing->stripe_checkout_session_id === $sessionId && $listing->status === 'active') {
          return;
      }

      $listing->update([
          'status'    => 'active',
          'listed_at' => now(),
      ]);

      Log::info("Listing {$listingId} activated via Stripe session {$sessionId}");
  }
  ```

  - Idempotent — safe to call multiple times for the same session

### §1 — Webhook Signature Verification is Mandatory

The Stripe webhook endpoint is public (no auth). Without signature verification, any attacker could POST a fake `checkout.session.completed` event to activate listings without payment. The `STRIPE_WEBHOOK_SECRET` (from Stripe Dashboard → Webhooks → Signing secret) must be:

1. Set in production `.env`
2. Set in local dev via Stripe CLI (`stripe listen --forward-to localhost:8000/api/v1/webhooks/stripe`)
3. Set to a test value in `phpunit.xml` for tests that verify signature rejection

Never skip the signature check. The `constructWebhookEvent` call in `StripeService` must ALWAYS verify the signature — no bypass in production.

---

### §2 — Idempotency for Webhooks

Stripe may deliver the same webhook event more than once (network retries). The `handleSuccessfulPayment` method is idempotent: it checks if the listing already has `status = 'active'` AND the same `stripe_checkout_session_id` before applying the update. A second delivery of the same event is a no-op.

This is critical — without idempotency, a double-delivered webhook could interfere with future state machines (e.g., a listing that's been refunded and deactivated could be re-activated).

### §4 — The `featured` vs `standard` Distinction at Checkout

The listing type (`standard` or `featured`) is determined at the time of `POST /api/v1/listings/{id}/payment/initiate`. The breeder does not select pricing on a Stripe page — they select it in the PawTrust UI (Story 2.2 listing form has a "Featured? +€5" option; with pay-per-listing enabled, the payment step confirms the type).

The `listing_type` is stored in the Stripe Checkout Session `metadata`. On webhook receipt, `metadata.listing_type` is used to determine if `is_featured` should be set to `true`. This story does not implement the upgrade path (Story 5.4); it only handles the initial standard/featured choice at creation time.
