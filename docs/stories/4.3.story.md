# Story 4.3: Post-Inquiry Document Access

**Epic:** 4 — Inquiry System & Contact Reveal
**Status:** Draft

---

## Story

**As a** buyer who has submitted an inquiry,
**I want** to access the listing's health and breeder documents,
**so that** I can review the pet's medical history and verify the breeder's credentials before proceeding.

---

## Acceptance Criteria

1. After submitting an inquiry, the buyer can access documents associated with the listing via `GET /api/v1/inquiries/{inquiry_id}/documents`. Auth required.
2. Server enforces that the requesting user is the buyer who submitted the inquiry — other buyers and unauthenticated users receive 403.
3. Response: list of documents, each with `{ id, document_type, filename }` and a time-limited pre-signed S3 URL (TTL: 1 hour) for download.
4. Documents are accessible as long as the inquiry record exists — no expiry on access rights post-inquiry.
5. Frontend: a "View Documents" button appears on the inquiry card in BuyerDashboard after an inquiry is submitted. Clicking triggers the API call and opens each document in a new browser tab.

    ```php
    public function getDocumentsForInquiry(Inquiry $inquiry): Collection
    {
        // Fetch breeder documents linked to this listing's breeder profile
        $documents = BreederDocument::where('breeder_profile_id', $inquiry->listing->breeder_profile_id)
            ->whereNull('deleted_at')
            ->get();

        return $documents->map(function (BreederDocument $doc) {
            return [
                'id'            => $doc->id,
                'document_type' => $doc->document_type,
                'filename'      => $doc->original_filename,
                'url'           => $this->s3->temporaryUrl($doc->s3_key, now()->addHour()),
                'expires_at'    => now()->addHour()->toISOString(),
            ];
        });
    }
    ```

## Dev Notes

### §1 — Document Access Authorization Model

The authorization check uses **ownership matching** (`inquiry.buyer_id === request.user().id`), not a role check. This means:

- A buyer can only access documents for their own inquiries
- Another buyer who happens to know the `inquiry_id` gets a 403
- A breeder or admin calling this endpoint also gets 403 (they don't own the inquiry)

This is intentional — the inquiry is the access grant. The document endpoint does not need a separate "document permission" table; the inquiry record IS the permission.

Do not add a `role:buyer` middleware to this route — it would incorrectly prevent an admin from ever accessing this endpoint if an admin-side document audit is needed in future. The ownership check is sufficient and more precise.

---

### §2 — Pre-Signed URL Generation in Tests

Tests use `Storage::fake('s3')`. The `Storage::fake()` disk does not support actual pre-signed URL generation (it throws or returns a local path). Handle this by:

1. Injecting the storage disk into `InquiryDocumentService` via constructor
2. In tests, mock the disk or mock the service method:
   ```php
   $mockDisk = $this->mock(FilesystemAdapter::class);
   $mockDisk->shouldReceive('temporaryUrl')->andReturn('https://s3.example.com/fake-url');
   ```
3. Assert that the returned `url` is a string starting with `https://` (not the s3_key)

The test for `test_s3_key_not_present_in_response()` should assert that the string value of `s3_key` from the document fixture does NOT appear anywhere in the JSON response body.

---

### §3 — Opening Multiple Tabs

Opening multiple browser tabs programmatically via `window.open()` in a loop may be blocked by some browsers as a pop-up. Acceptable mitigations (all within scope):

- If only 1 document: `window.open` fires once — no issue
- If multiple documents: use a small sequential delay (50ms per document) or render a modal listing documents with direct download links instead of auto-opening tabs

For MVP, if the listing has more than 1 document, render the documents list in a modal (`DocumentListModal`) with one `<a target="_blank">` link per document rather than calling `window.open` in a loop. This avoids browser popup blocking entirely and is a better UX pattern.

Update `InquiryDocumentsButton.tsx` accordingly: on success with exactly 1 document, open directly; on success with 2+ documents, show a modal.

```bash
curl -s "http://localhost:8000/api/v1/inquiries/$INQUIRY_ID/documents" \

# Confirm s3_key is absent

# → false
```
