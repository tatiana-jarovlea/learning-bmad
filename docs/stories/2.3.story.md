# Story 2.3: Breeder Document Upload (Profile Documents)

**Epic:** 2 — Breeder Profiles & Listings
**Status:** Draft

---

## Story

**As a** breeder,
**I want** to upload my credentials and certifications to my profile,
**so that** buyers can see I have documents on file and admins can verify my legitimacy.

---

## Acceptance Criteria

1. Breeders can upload documents to their profile: cynology federation membership certificate, FCI/AChR registration papers, vaccination records, breed health test results. Each upload includes a `document_type` label.
2. Upload endpoint: `POST /api/v1/breeders/{id}/documents` — multipart form with `file` + `document_type`. Auth required, breeder role only; breeder can only upload to their own profile.
3. Files stored in private S3 bucket. Max 10MB per document. PDF and image (JPEG/PNG/WebP) formats accepted. File type validated server-side via magic bytes, not just MIME type.
4. Documents listed in the authenticated breeder's own profile response as: `[{ id, document_type, filename, uploaded_at, status }]` — no S3 URL or key returned.
5. Admin can access pre-signed S3 URLs for documents via `GET /api/v1/breeders/{id}/documents/{docId}/url` (admin role only). URL TTL: 15 minutes.
6. Breeder can soft-delete their own documents via `DELETE /api/v1/breeders/{id}/documents/{docId}`. Soft-deleted documents set `deleted_at` (retained in S3 for 30 days for audit). They no longer appear in the breeder's document list.
7. Document count shown on public breeder profile: `"documents_on_file": 3` — no filenames, types, or URLs revealed to anonymous users.

## Dev Notes

### §1 — Soft Delete vs `deleted` Enum Status

The architecture schema defines `status` as `enum('pending','approved','rejected')` with no `deleted` value. AC6 requires 30-day audit retention. The chosen pattern:

- Add `deleted_at` (soft delete) via `SoftDeletes` trait — this is the standard Laravel pattern for auditable deletions
- Soft-deleted documents are excluded from the breeder's document list (default Eloquent behaviour with `SoftDeletes`)
- They remain physically in S3 and in the `breeder_documents` table with a populated `deleted_at`
- A scheduled job to purge S3 objects where `deleted_at < NOW() - 30 days` is OUT OF SCOPE for MVP — documented as a future technical debt item
- Do NOT add a `deleted` status to the enum — mixing deletion state with review state is a design smell

### §3 — Security: 30-Day Audit Retention

Calling `$document->delete()` in Laravel with `SoftDeletes` sets `deleted_at` but leaves the record in the DB and the file in S3. This is intentional:

1. **Audit trail:** If a breeder uploads a fraudulent document and then deletes it to cover their tracks, the admin can still see it was uploaded (via DB record with `deleted_at`).
2. **S3 retention:** The S3 object stays until a scheduled purge job runs. This job is out of scope for MVP — add as a maintenance story after launch.

A future `PurgeDeletedDocumentsJob` should: find all `breeder_documents` where `deleted_at < now()->subDays(30)` → delete S3 object → hard-delete the DB row.

---

### §4 — Document Types: Defined Enum vs Free String

`document_type` is validated as an `in:` rule on the server — it is NOT a free-form string. This prevents:

- Junk data in the column
- Inconsistent type labels in admin UI

Valid values at MVP: `kennel_cert`, `fci_papers`, `achr_papers`, `vaccination_records`, `health_tests`, `other`.
Adding new types requires a backend deploy (acceptable for MVP; configurable in a future admin settings story).
